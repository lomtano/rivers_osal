# mem：统一堆与内存池

## 1. 模块职责

`osal_mem` 提供两类能力：

- 统一静态堆分配
- 固定块内存池

对应文件：

- [osal_mem.h](/abs/path/A:/Embedded_system/cubemx_project/rivers_osal/Middleware/osal/system/Inc/osal_mem.h)
- [osal_mem.c](/abs/path/A:/Embedded_system/cubemx_project/rivers_osal/Middleware/osal/system/Src/osal_mem.c)

## 2. 统一静态堆的定位

当前 OSAL 不依赖系统 `malloc/free`，而是自己维护一块静态堆。

它承担的主要职责是：

- 任务控制块
- 队列控制块
- 事件对象
- 互斥量对象
- 软件定时器条目
- 组件对象

也就是说，它更像一个内核对象堆，而不是通用 C 库堆。

## 3. 统一堆的数据结构

统一堆的空闲块头部一般包含：

- `size_and_flags`
- `next_free`

其中：

- 高位通常表示“是否已占用”
- 低位表示块大小

这样做的好处是：

- 元数据紧凑
- 释放和合并逻辑简单

## 4. 分配流程

统一堆分配大体是：

1. 对齐请求大小
2. 加上块头开销
3. 从空闲链表里找第一个足够大的块
4. 如果剩余空间足够，再拆分成：
   - 已分配块
   - 新空闲块
5. 返回用户数据区地址

这是典型的首适配策略。

## 5. 为什么要按地址顺序插回空闲块

释放时，空闲块不是随便塞回链表，而是按地址顺序插入。

原因是：

- 只有知道前后物理地址关系
- 才能判断是不是和相邻块连续
- 进而做块合并

如果不按地址顺序组织空闲链表，合并逻辑会更复杂。

## 6. 合并的意义

释放后如果只把块放回去、不合并，长期运行会出现：

- 空闲总量不少
- 但被碎成很多小块
- 大块分配反而失败

所以合并的作用是降低碎片。

## 7. 当前统一堆的边界

这套堆是裸机轻量实现，不是通用高并发 `malloc`。

边界包括：

- 更适合内核对象分配
- 不追求复杂最佳适配策略
- 主要靠简单、稳定、可控

## 8. 固定块内存池的定位

除了统一堆，当前还支持 mempool。

固定块内存池适合：

- 高频创建/释放
- 对象大小固定
- 需要更稳定、更快的分配开销

例如：

- 固定大小消息块
- 固定大小缓存块

## 9. 内存池如何组织

内存池要求用户给一块连续缓冲区。

初始化时：

- 按 `block_size` 和对齐规则算出步长 `stride`
- 把每个块串成单向空闲链表

分配时：

- 从空闲链表表头摘一个块

释放时：

- 把块头重新插回空闲链表

所以它非常轻量。

## 10. 统一堆和内存池的区别

### 统一堆

- 灵活
- 适合大小不一致的对象
- 有碎片管理逻辑

### 固定块池

- 更快
- 更稳定
- 适合固定大小对象
- 但灵活性低

## 11. RAM 开销该怎么看

当前这套 OSAL 的 RAM 大头通常不是任务调度器本身，而是：

- `OSAL_HEAP_SIZE`

所以评估内存占用时要分开看：

1. 固定全局状态开销
2. 统一堆大小
3. 运行期对象占用

## 12. 阅读源码时建议关注

建议重点看：

- 块头定义
- 对齐逻辑
- 分裂逻辑
- 空闲链表插回逻辑
- 合并逻辑
- mempool 的 `stride` 和 free-list 初始化

这些部分决定了内存系统的稳定性。
