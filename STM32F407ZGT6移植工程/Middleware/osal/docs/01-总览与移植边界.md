# OSAL 总览与移植边界

## 1. 定位

当前 `OSAL` 是一套面向 Cortex-M 裸机工程的纯协作式内核。
它保留的基础能力只有这些：

- `task`：按优先级决定运行顺序的协作式任务调度
- `timer`：SysTick 计时、`us/ms` 忙等延时、软件定时器
- `queue`：固定单元大小的环形消息队列，任务态保留同步 `timeout_ms` 重试接口
- `mem`：基于静态内存池的动态分配算法，主要服务于 queue 等内核对象
- `irq`：对 CMSIS 中断开关接口的轻量封装
- `cortexm`：SysTick、NVIC Group (4)、DWT 等 Cortex-M 内核外设配置

`event` 和 `mutex` 已从当前公开接口与工程编译单元中移除。

## 2. 模块边界

### 2.1 保留模块

- `task`
- `timer`
- `irq`
- `mem`
- `queue`
- `cortexm`

### 2.2 已删除模块

- `event`
- `mutex`

### 2.3 分层职责

- `system`
  负责 OSAL 内核本身。
- `cortexm`
  负责内核真正依赖的 Cortex-M 外设和寄存器配置。
- `platform/example`
  负责板级示例桥接，例如 LED、USART、Flash。
- `components`
  放在 OSAL 之上的外围组件。

## 3. 公共入口与配置

### 3.1 `osal_config.h`

`system/Inc/osal_config.h` 是统一配置入口，当前集中承载：

- `OSAL_CFG_ENABLE_DEBUG`
- `OSAL_CFG_ENABLE_QUEUE`
- `OSAL_CFG_ENABLE_IRQ_PROFILE`
- `OSAL_CFG_ENABLE_SW_TIMER`
- `OSAL_CFG_ENABLE_USART`
- `OSAL_CFG_ENABLE_FLASH`
- `OSAL_CFG_INCLUDE_PLATFORM_HEADER`
- `OSAL_PLATFORM_HEADER_FILE`

### 3.2 `osal.h`

`osal.h` 的职责已经收敛为：

1. 包含 `osal_config.h`
2. 定义统一状态码
3. 提供调试钩子宏
4. 聚合各模块头文件

## 4. 初始化与运行流程

推荐接入顺序如下：

1. 完成板级硬件初始化
2. 调用 `osal_init()`
3. 创建并启动任务、组件或软件定时器
4. 在主循环中持续调用 `osal_run()`
5. 在 `SysTick_Handler()` 中调用 `osal_tick_handler()`

`osal_init()` 当前会依次执行：

1. `osal_cortexm_init()`
2. `osal_cortexm_setup_interrupt_controller()`
3. `osal_cortexm_profile_init()`
4. `osal_cortexm_setup_system_tick()`
5. `osal_timer_sync_tick_source()`

## 5. 真实执行模型

### 5.1 task

- 任务没有独立任务栈。
- 任务没有上下文保存/恢复。
- 调度器仍然按优先级挑选当前可运行任务。
- 任务函数本质上是“被调度器反复调用的普通 C 函数”。

### 5.2 yield

- `osal_task_yield()` 会在当前调用栈里同步触发一次嵌套调度。
- 嵌套调度返回后，当前任务从 `yield()` 的下一条语句继续执行。
- 它不是挂起，不是恢复，也不是切换到另一条独立执行栈。

### 5.3 queue

- `queue` 本质上可以看成“使用 `osal_mem` 动态分配底层存储的固定单元环形缓冲区”。
- `osal_queue_send/recv(..., 0U)` 立即尝试一次。
- `osal_queue_send/recv(..., N)` 会在 `N ms` 窗口内基于系统 tick 同步忙等重试。
- 当前 queue 没有等待链表，也不会在 send/recv 后把某个等待任务切回 `READY`。
- 它不保存任务上下文，也不支持“恢复到中断前正在等待 queue 的那一行继续执行”。
- 任务侧如果要等消息，本质上仍然要自己安排下一次检查时机。

### 5.4 timer

- `delay_us()` / `delay_ms()` 是同步忙等。
- 软件定时器通过 `osal_timer_poll()` 在主循环中推进。

## 6. 适用场景

适合：

- 裸机主循环 + 协作任务
- 任务函数短小、业务可写成状态机
- 队列状态会被 ISR、DMA 或其他异步硬件路径推进

不适合：

- 需要抢占式调度
- 需要独立任务栈
- 需要真正的阻塞等待和恢复语义
- 希望用 `queue(timeout_ms)` 代替任务层状态机或等待/唤醒机制

## 7. 移植时主要关注点

移植到新板子时，优先关注这些配置：

- `OSAL_CORTEXM_CPU_CLOCK_HZ`
- `OSAL_CORTEXM_SYSTICK_CLOCK_HZ`
- `OSAL_CORTEXM_TICK_RATE_HZ`
- `OSAL_CORTEXM_HAS_DWT_CYCCNT`
- LED / USART / Flash 这类板级桥接宏与实现

核心原则是：

- SysTick、IRQ、DWT 这类 OSAL 内核真正依赖的底层能力放在 `system/cortexm`。
- 板级外设桥接放在 `platform/example/<board>`。
