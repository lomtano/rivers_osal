# OSAL 总览与移植步骤

## 1. 定位

当前 `OSAL` 是一套面向 Cortex-M 裸机工程的纯协作式内核。
它保留的基础能力只有这些：

- `task`：按优先级决定运行顺序的协作式任务调度
- `timer`：SysTick 计时、`us/ms` 忙等延时、软件定时器
- `queue`：固定单元大小的环形消息队列，任务态保留同步 `timeout_ms` 重试接口
- `mem`：基于静态内存池的动态分配算法，主要服务于 queue 等内核对象
- `irq`：对 CMSIS 中断开关接口的轻量封装
- `cortexm`：SysTick、NVIC Group (4)、DWT 等 Cortex-M 内核外设配置

`event` 和 `mutex` 已从当前公开接口与工程编译单元中移除。

## 2. 模块边界

### 2.1 保留模块

- `task`
- `timer`
- `irq`
- `mem`
- `queue`
- `cortexm`

### 2.2 已删除模块

- `event`
- `mutex`

### 2.3 分层职责

- `system`
  负责 OSAL 内核本身。
- `cortexm`
  负责内核真正依赖的 Cortex-M 外设和寄存器配置。
- `platform/example`
  负责板级示例桥接，例如 LED、USART、Flash。
- `components`
  放在 OSAL 之上的外围组件。

## 3. 公共入口与配置

### 3.1 `osal_config.h`

`system/Inc/osal_config.h` 是统一配置入口，当前集中承载：

- `OSAL_CFG_ENABLE_DEBUG`
- `OSAL_CFG_ENABLE_QUEUE`
- `OSAL_CFG_ENABLE_IRQ_PROFILE`
- `OSAL_CFG_ENABLE_SW_TIMER`
- `OSAL_CFG_ENABLE_USART`
- `OSAL_CFG_ENABLE_FLASH`
- `OSAL_CFG_INCLUDE_PLATFORM_HEADER`
- `OSAL_PLATFORM_HEADER_FILE`

### 3.2 `osal.h`

`osal.h` 的职责已经收敛为：

1. 包含 `osal_config.h`
2. 定义统一状态码
3. 提供调试钩子宏
4. 聚合各模块头文件

## 4. 初始化与运行流程

推荐接入顺序如下：

1. 完成板级硬件初始化
2. 调用 `osal_init()`
3. 创建并启动任务、组件或软件定时器
4. 在主循环中持续调用 `osal_run()`
5. 在 `SysTick_Handler()` 中调用 `osal_tick_handler()`

`osal_init()` 当前会依次执行：

1. `osal_cortexm_init()`
2. `osal_cortexm_setup_interrupt_controller()`
3. `osal_cortexm_profile_init()`
4. `osal_cortexm_setup_system_tick()`
5. `osal_timer_sync_tick_source()`

## 5. 真实执行模型

### 5.1 task

- 任务没有独立任务栈。
- 任务没有上下文保存/恢复。
- 调度器仍然按优先级挑选当前可运行任务。
- 任务函数本质上是“被调度器反复调用的普通 C 函数”。

### 5.2 yield

- `osal_task_yield()` 会在当前调用栈里同步触发一次嵌套调度。
- 嵌套调度返回后，当前任务从 `yield()` 的下一条语句继续执行。
- 它不是挂起，不是恢复，也不是切换到另一条独立执行栈。

### 5.3 queue

- `queue` 本质上可以看成“使用 `osal_mem` 动态分配底层存储的固定单元环形缓冲区”。
- `osal_queue_send/recv(..., 0U)` 立即尝试一次。
- `osal_queue_send/recv(..., N)` 会在 `N ms` 窗口内基于系统 tick 同步忙等重试。
- 当前 queue 没有等待链表，也不会在 send/recv 后把某个等待任务切回 `READY`。
- 它不保存任务上下文，也不支持“恢复到中断前正在等待 queue 的那一行继续执行”。
- 任务侧如果要等消息，本质上仍然要自己安排下一次检查时机。

### 5.4 timer

- `delay_us()` / `delay_ms()` 是同步忙等。
- 软件定时器通过 `osal_timer_poll()` 在主循环中推进。

## 6. 适用场景

适合：

- 裸机主循环 + 协作任务
- 任务函数短小、业务可写成状态机
- 队列状态会被 ISR、DMA 或其他异步硬件路径推进

不适合：

- 需要抢占式调度
- 需要独立任务栈
- 需要真正的阻塞等待和恢复语义
- 希望用 `queue(timeout_ms)` 代替任务层状态机或等待/唤醒机制

## 7. 移植步骤清单

下面这份清单按“先能跑起来，再逐步裁剪”的顺序写。

### 7.1 第一步：先决定要带哪些文件

最省事的接入方式是：

1. 把 `Middleware/osal/system/Inc` 全部头文件加入包含路径。
2. 把 `Middleware/osal/system/Src` 里的当前源文件加入工程：
   - `osal_task.c`
   - `osal_timer.c`
   - `osal_mem.c`
   - `osal_irq.c`
   - `osal_cortexm.c`
   - `osal_queue.c`
3. 如果要用 `USART / Flash` 组件，再把对应 `components/periph/...` 文件加入工程。
4. 如果只是移植 OSAL 内核本身，而不是复用当前 STM32F4 示例桥接，就不要把 `platform/example/stm32f4` 当成必须依赖。

推荐先把 `system` 层整体加进去，等功能跑通后再按 `osal_config.h` 做裁剪。

### 7.2 第二步：先改 `osal_config.h`

`system/Inc/osal_config.h` 是第一层裁剪入口。移植到新板子时，先把这些开关确认一遍：

- `OSAL_CFG_ENABLE_DEBUG`
  调试阶段建议先开，方便尽早暴露错误上下文调用和非法句柄。
- `OSAL_CFG_ENABLE_QUEUE`
  需要消息队列就保持 `1`，完全不用队列就改成 `0`。
- `OSAL_CFG_ENABLE_IRQ_PROFILE`
  只有你确实要看 DWT 临界区统计时才开；大多数初次移植建议先关。
- `OSAL_CFG_ENABLE_SW_TIMER`
  需要软件定时器就开，不需要就关。
- `OSAL_CFG_ENABLE_USART`
  只有把 `periph_uart` 组件一起带进工程时才保持 `1`。
- `OSAL_CFG_ENABLE_FLASH`
  只有把 `periph_flash` 组件一起带进工程时才保持 `1`。
- `OSAL_CFG_INCLUDE_PLATFORM_HEADER`
  如果你不想让 `osal.h` 自动包含板级示例头文件，改成 `0`。
- `OSAL_PLATFORM_HEADER_FILE`
  如果你保留“自动包含平台头文件”这条路径，就把它改成你自己的板级头文件名。

一个很实用的经验是：

- 如果你只是先移植 `task / timer / mem / queue / irq / cortexm` 这套内核，通常先把 `OSAL_CFG_INCLUDE_PLATFORM_HEADER` 设成 `0`。
- 如果你还没把 `periph_uart`、`periph_flash` 加进工程，就把 `OSAL_CFG_ENABLE_USART`、`OSAL_CFG_ENABLE_FLASH` 一并关掉。

### 7.3 第三步：改 `osal_cortexm.h`

`system/Inc/osal_cortexm.h` 是第二层移植入口，负责 OSAL 真正依赖的 Cortex-M 内核外设配置。

先改 `SysTick` 这组：

- `OSAL_CORTEXM_CPU_CLOCK_HZ`
  改成目标板当前真正的 CPU 主频。
- `OSAL_CORTEXM_SYSTICK_CLOCK_HZ`
  改成 SysTick 实际输入时钟；如果 SysTick 跟 CPU 同频，通常保持等于 `CPU_CLOCK_HZ`。
- `OSAL_CORTEXM_TICK_RATE_HZ`
  设定 OSAL 主 tick 频率；当前默认是 `1000`，也就是 `1 ms` 一拍。
- `OSAL_CORTEXM_SYSTICK_USE_CORE_CLOCK`
  如果 SysTick 用的是内核时钟 `HCLK`，保持 `1`；如果你明确切成了分频时钟，再改成 `0`。
- `OSAL_CORTEXM_SYSTICK_HANDLER_NAME`
  如果你的启动文件里系统时基中断函数名不是 `SysTick_Handler`，就在这里改名。

再改 `NVIC Group (4)` 这组：

- `OSAL_CORTEXM_CONFIGURE_PRIORITY_GROUP`
  如果想让 OSAL 自己写 `AIRCR.PRIGROUP`，保持 `1`；如果这件事已经由启动代码或 HAL 处理，改成 `0`。
- `OSAL_CORTEXM_NVIC_PRIORITY_GROUP_RAW`
  当前默认 `3U`，对 STM32F4 对应常见的 `NVIC Group 4`。
- `OSAL_CORTEXM_CONFIGURE_SYSTICK_PRIORITY`
  如果想让 OSAL 自己设置 SysTick 优先级，保持 `1`；否则改成 `0`。
- `OSAL_CORTEXM_NVIC_PRIO_BITS`
  改成目标内核实际实现的优先级位数。
- `OSAL_CORTEXM_SYSTICK_PRIORITY_LEVEL`
  改成你想给 SysTick 的优先级档位；当前默认是最低优先级。

最后确认 `DWT` 和 `IRQ` 这组：

- `OSAL_CORTEXM_HAS_DWT_CYCCNT`
  如果目标内核不支持 DWT CYCCNT，例如常见 `Cortex-M0/M0+`，改成 `0`。
- `OSAL_CORTEXM_DWT_HAS_LAR`
  只有目标内核实现了 DWT LAR 解锁寄存器时才改成 `1`。
- `OSAL_CORTEXM_IRQ_GET_IPSR`
- `OSAL_CORTEXM_IRQ_GET_PRIMASK`
- `OSAL_CORTEXM_IRQ_RAW_DISABLE`
- `OSAL_CORTEXM_IRQ_RAW_ENABLE`

上面这四个 `IRQ/CMSIS` 宏默认直接对接 CMSIS。如果你的编译器、SDK 或 CMSIS 头文件路径不兼容默认实现，就在移植时显式覆盖它们。

### 7.4 第四步：把 OSAL 接进运行流程

移植完成后，工程里至少要接上这三个运行点：

1. 板级时钟、GPIO、串口这类硬件初始化完成后，调用一次 `osal_init()`。
2. 在主循环里持续调用 `osal_run()`。
3. 在系统时基中断里调用 `osal_tick_handler()`。

最小骨架就是：

```c
int main(void) {
    board_init();
    osal_init();

    for (;;) {
        osal_run();
    }
}

void SysTick_Handler(void) {
    osal_tick_handler();
}
```

如果你的工程已经有自己的 `SysTick_Handler()` 内容，就把 `osal_tick_handler()` 合进去，而不是再新建一个平行的时基中断入口。

当前默认模型下，主循环持续调用 `osal_run()` 就够了，因为 `osal_run()` 内部已经会先推进一次 `osal_timer_poll()`。

### 7.5 第五步：按这个顺序做自检

建议不要一上来就把所有组件都带起来，先做最小自检：

1. 确认 `osal_init()` 执行后没有触发明显的配置错误钩子。
2. 确认 `SysTick` 中断确实在按预期频率进入，`osal_tick_handler()` 被周期调用。
3. 确认主循环里的 `osal_run()` 在持续执行，没有被别的阻塞逻辑卡死。
4. 如果开了 `queue`，先测一组最简单的 `send/recv(..., 0U)`。
5. 如果开了软件定时器，再测 `osal_timer_poll()` 推动下的回调是否按时执行。
6. 如果开了 `OSAL_CFG_ENABLE_IRQ_PROFILE`，再去看 DWT profiling 是否可用；不要把 profiling 当成最初移植是否成功的前置条件。

### 7.6 迁移边界不要放错地方

最后记住这条边界：

- SysTick、IRQ、DWT、CMSIS 宏映射这类“OSAL 内核真正依赖的底层能力”，放在 `system/cortexm`。
- LED、USART、Flash 这类“板级外设桥接”，放在 `platform/example/<board>` 或你自己的板级目录。
- 如果只是想把 OSAL 内核移植到新板子，不需要先把现有 `platform/example/stm32f4` 一起照搬过去。
