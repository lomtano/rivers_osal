# 总览与移植边界

## 1. 目录分层

这套 `OSAL` 当前分成三层：

- `system`
  - 内核层
  - 包含任务调度、时基、消息队列、内存、IRQ、事件、互斥量等核心实现
- `platform`
  - 架构与板级桥接层
  - `system` 通过这里接管 `SysTick` 和 `IRQ` 宏
  - `example/stm32f4` 里放当前板子的具体桥接实现和集成示例
- `components`
  - 挂在 OSAL 上的外围组件
  - 当前包括 `periph/usart`、`periph/flash`、`RTT`、`KEY`

简化理解：

- `system` 解决“内核怎么工作”
- `platform` 解决“内核怎么接上 Cortex-M 和具体 MCU SDK”
- `components` 解决“在内核之上怎么挂串口、Flash、RTT、按键等能力”

## 2. 当前系统的性质

这套 OSAL 不是抢占式 RTOS，而是：

- 基于 `SysTick` 的统一时基
- 基于协作式任务模型的轻量调度
- 基于事件驱动队列的消息同步
- 基于统一静态堆的对象管理

它更像一个“带任务、时基和 IPC 的裸机逻辑框架”，而不是 FreeRTOS 的直接替代品。

## 3. 启动链路

典型启动顺序如下：

1. `main()` 完成芯片和外设基础初始化
2. 调用 `osal_init()`
3. `osal_init()` 进入 `system` 层平台初始化逻辑
4. `system` 自动配置 `NVIC`、`SysTick`、原始 tick source
5. 应用层在 `SysTick_Handler()` 中调用 `osal_tick_handler()`
6. 主循环中持续调用 `osal_run()`

运行时主链路可以概括为：

- `osal_tick_handler()` 负责累加基础时间
- `osal_run()` 负责：
  - 轮询软件定时器
  - 调度高/中/低优先级任务
  - 处理被唤醒或超时恢复的任务

## 4. 什么地方算“内核”

当前内核核心主要是下面几个文件：

- `osal_task.c`
- `osal_timer.c`
- `osal_queue.c`
- `osal_mem.c`
- `osal_irq.c`
- `osal_platform.c`

这几个文件共同决定：

- 任务如何调度
- 时间如何累计
- 软件定时器如何工作
- 队列如何等待和唤醒
- 内存如何分配和回收
- `SysTick / IRQ` 如何接入

## 5. 当前真正的移植边界

这套系统并不是“完全与所有 MCU 解耦”，更准确的说法是：

- 与厂商 SDK 解耦
- 对 Cortex-M 架构有依赖

依赖点集中在：

- `osal_platform.h`
- `osal_platform.c`

这里面用了：

- Cortex-M `SysTick` 寄存器
- `NVIC / AIRCR / SHPR3`
- CMSIS 风格的中断宏

所以：

- STM32 / GD32 / N32 这类 Cortex-M MCU，正常移植成本较低
- 非 Cortex-M 架构，必须先重写 platform 抽象

## 6. 正常移植时应该改哪里

从一个 Cortex-M 平台移植到另一个 Cortex-M 平台时，原则上应主要改：

- `osal/system/Inc/osal_platform.h`
  - CPU 频率
  - SysTick 频率
  - tick 频率
  - IRQ 宏
  - NVIC/SysTick 相关宏
- `osal/platform/example/<board>/osal_platform_xxx.c`
- `osal/platform/example/<board>/osal_platform_xxx.h`

不应该优先去改：

- `task`
- `queue`
- `timer`
- `mem`

如果移植一个新 MCU 时需要频繁改这些内核文件，说明边界设计出了问题。

## 7. 当前实现里的几个重要事实

### 7.1 任务是协作式，不是抢占式

高优先级的含义是“先被调度器检查”，不是“立刻打断别的任务”。

### 7.2 队列已经是事件驱动等待

当前队列等待不是简单轮询，而是：

- 任务进入 `BLOCKED`
- 挂到等待发送/等待接收链表
- `send/recv/from_isr` 成功后主动唤醒相关任务

### 7.3 事件和互斥量还不是同等级实现

`event` 和 `mutex` 目前仍然是：

- 条件不满足时 `yield`
- 不是像队列那样的等待链表 + 事件唤醒模型

### 7.4 忙等延时仍然会占 CPU

下面这类接口本质上是忙等：

- `osal_timer_delay_us()`
- `osal_timer_delay_ms()`

它们适合短延时，不适合长时间滥用。

## 8. 阅读源码的推荐顺序

建议按下面顺序读：

1. `osal_task.c`
2. `osal_timer.c`
3. `osal_queue.c`
4. `osal_mem.c`
5. `osal_platform.c`
6. `osal_irq.c`
7. `osal_event.c`
8. `osal_mutex.c`
9. `periph_uart.c`
10. `periph_flash.c`

原因是前四个文件决定了这套系统的主行为。
